# k8s-irsa-without-eks

This repository provides a setup to enable IRSA on a Kubernetes cluster built with kubeadm (or similar), without using EKS.

In this repository, you will mainly:

- Expose OIDC Discovery (`/.well-known/openid-configuration`) and JWKS (`/openid/v1/jwks`)
- Create Route53/domain/DNS resources for a public endpoint
- Install cert-manager, Envoy Gateway, aws-cloud-controller-manager, and amazon-eks-pod-identity-webhook
- Create an AWS IAM OIDC Provider and IAM Roles for Kubernetes ServiceAccounts

## Prerequisites

- Terraform is available.
- You can access the target Kubernetes cluster with `kubectl` (target context selected in advance).
- The EC2 instance running the Kubernetes cluster has an Elastic IP (EIP).
- Port `443/TCP` is available.
- Terraform in this repository assumes local state (remote backend/state management is not supported).

## Overall Execution Order

1. Apply `terraform/route53`
2. Apply `helmfiles`
3. Install/upgrade `helm-chart`
4. Update kube-apiserver `issuer` / `jwks_uri` to the public FQDN
5. Apply `terraform/oidc`

## 1. terraform/route53

Creates Route53-related resources and also generates `helm-chart/values.generated.yaml`.

### 1-1. Prepare tfvars

```bash
cp terraform/route53/terraform.tfvars.sample terraform/route53/terraform.tfvars
```

Main fields in `terraform/route53/terraform.tfvars`:

- `var_provider.region`: AWS region
- `route53domains_domain`: Domain settings to register/manage (including contact information)
- `iam_user.name`: IAM user name used by cert-manager for Route53 DNS-01
- `iam_policy.name`: Policy name attached to that IAM user
- `route53_a_record_name`: Public FQDN to expose (for example, `app.example.com`)
- `route53_a_record_ip`: Global IP assigned to that FQDN (for example, EIP)

### 1-2. Run

```bash
terraform -chdir=terraform/route53 init
terraform -chdir=terraform/route53 plan
terraform -chdir=terraform/route53 apply
```

After `apply`, `helm-chart/values.generated.yaml` is updated.

### 1-3. Check outputs (optional)

```bash
terraform -chdir=terraform/route53 output
```

## 2. helmfiles

Installs required components in the Kubernetes cluster:

- cert-manager
- Envoy Gateway
- amazon-eks-pod-identity-webhook
- aws-cloud-controller-manager

### 2-1. Prepare vars files

Copy from example/sample files as needed and edit values.

```bash
cp helmfiles/vars/cert-manager.yaml.example helmfiles/vars/cert-manager.yaml
cp helmfiles/vars/envoy-gateway.yaml.sample helmfiles/vars/envoy-gateway.yaml
cp helmfiles/vars/aws-cloud-controller-manager.yaml.example helmfiles/vars/aws-cloud-controller-manager.yaml
cp helmfiles/vars/amazon-eks-pod-identity-webhook.yaml.example helmfiles/vars/amazon-eks-pod-identity-webhook.yaml
```

### 2-2. Apply all four releases

```bash
helmfile -f helmfiles/helmfile.yaml apply
```

### 2-3. Apply only one release (recommended)

```bash
release="cert-manager"
helmfile -f helmfiles/helmfile.yaml -l name="${release}" apply
```

`release` can be one of:

- `cert-manager`
- `eg`
- `aws-cloud-controller-manager`
- `amazon-eks-pod-identity-webhook`

### 2-4. Apply only one release (direct release file)

```bash
release="cert-manager"
helmfile \
  -f helmfiles/releases/"${release}".yaml.gotmpl \
  --state-values-file ../vars/"${release}".yaml \
  apply
```

`release` can be one of:

- `cert-manager`
- `envoy-gateway`
- `amazon-eks-pod-identity-webhook`
- `aws-cloud-controller-manager`

## 3. helm-chart

Deploys resources to expose kube-apiserver OIDC Discovery/JWKS over HTTPS.

Uses `helm-chart/values.generated.yaml`, generated by `terraform/route53`.

### 3-1. Initial install

```bash
helm install k8s-irsa-without-eks ./helm-chart \
  -f helm-chart/values.generated.yaml
```

### 3-2. Upgrade after first install

```bash
helm upgrade k8s-irsa-without-eks ./helm-chart \
  -f helm-chart/values.generated.yaml
```

## 4. Update kube-apiserver issuer / jwks_uri

To make external services (AWS STS) interpret OIDC metadata correctly, update kube-apiserver to return the public FQDN in `issuer` and `jwks_uri`.

### 4-1. Required change

On the node running kube-apiserver, update the following options in `/etc/kubernetes/manifests/kube-apiserver.yaml`:

```diff
- --service-account-issuer=https://kubernetes.default.svc.cluster.local
+ --service-account-issuer=https://[YOUR_FQDN]
+ --service-account-jwks-uri=https://[YOUR_FQDN]/openid/v1/jwks
```

After the change, wait for kube-apiserver to restart and become Ready.

#### Notes

Some pods (for example, `cilium-operator` in CNI setups) may fail to start after this change. If that happens, restarting the affected Deployment/DaemonSet should resolve it.

### 4-2. Verify

```bash
curl https://[YOUR_FQDN]/.well-known/openid-configuration 2>/dev/null | jq
```

Expected output:

```json
{
  "issuer": "https://[YOUR_FQDN]",
  "jwks_uri": "https://[YOUR_FQDN]/openid/v1/jwks"
}
```

Additional notes:

- Anonymous access permissions for OIDC publication (ClusterRole/ClusterRoleBinding) are created by the Helm chart.
- If you place a backup of `kube-apiserver.yaml` in the same directory, it may be misrecognized as an additional static pod manifest.

## 5. terraform/oidc

Creates an AWS OIDC Provider and IAM Roles associated with Kubernetes ServiceAccounts.

### 5-1. Prepare tfvars

```bash
cp terraform/oidc/terraform.tfvars.sample terraform/oidc/terraform.tfvars
```

Main fields in `terraform/oidc/terraform.tfvars`:

- `oidc_provider.url`: Public OIDC issuer URL (for example, `https://app.example.com`)
- `oidc_provider.client_id_list`: Usually `"sts.amazonaws.com"`
- `oidc_assume_roles`: List of IAM roles to create
  - `namespace`
  - `service_account`
  - `audience` (usually `sts.amazonaws.com`)
  - `managed_policy_arns` (optional)
  - `inline_policy_json` (optional)

### 5-2. Run

```bash
terraform -chdir=terraform/oidc init
terraform -chdir=terraform/oidc plan
terraform -chdir=terraform/oidc apply
```

### 5-3. Check outputs (optional)

```bash
terraform -chdir=terraform/oidc output
```
